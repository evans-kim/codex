## 1. 목표와 범위

아내가 현재 기분, 체력, 당기는 음식의 느낌, 식사 방식을 선택하면 저녁 메뉴 세 가지를 이미지와 함께 추천하는 정적 웹앱을 구현한다. 모든 계산과 개인화 기록은 브라우저 안에서만 처리한다.

제외 범위는 예산 입력, 서버·데이터베이스, 회원가입, 외부 AI·음식점 API, 위치 정보, 광고·분석 도구다. 앱은 심리 상태를 진단하지 않고 사용자가 직접 고른 현재 기분을 메뉴 선택의 힌트로만 사용한다.

## 3. 페이즈별 상세 작업

### Phase 0 — 범위와 아키텍처

- Vanilla JavaScript ES Modules, HTML5, CSS3만 사용
- 서버 호출 없이 모든 추천 계산을 클라이언트에서 수행
- 사용자 데이터 저장소는 `localStorage` 한 종류로 단순화
- 외부 폰트·이미지·CDN·분석 요청을 금지하는 CSP 적용
- GitHub Pages 프로젝트 사이트의 하위 경로에서도 동작하도록 모든 URL을 상대 경로로 구성

### Phase 1 — 핵심 사용자 흐름

- 시작 화면과 빠른 추천
- 기분 → 체력 → 음식 느낌(최대 2개) → 식사 방식의 4단계 흐름
- 선택 상태, 진행률, 이전 단계 이동
- 대표 추천 1개와 대안 2개
- 메뉴 확정, 재추천, 처음부터 다시 선택
- 모바일 1열과 넓은 화면 2열 레이아웃

### Phase 2 — 추천 엔진과 이미지

- 한식·일식·중식·아시아·양식·분식·야식 등 메뉴 30개
- 기분·체력·맛·식사 방식·추가 조건별 가중치
- 예민한 날의 자극 감점, 무기력한 날의 배달·포장 가점 등 교차 규칙
- 상위 후보군에서 카테고리를 분산하는 다양화 로직
- 답변과 메뉴 특성을 연결한 추천 이유 생성
- 프로젝트 전용 SVG 음식 일러스트 30장과 기본 이미지

### Phase 3 — 개인화와 데이터 권리

- 좋아요·별로예요에 따른 메뉴별 선호 점수
- 최근 0·3·7·14일 메뉴 감점 설정
- 최근 식사 60건 저장 및 기록 화면
- JSON 백업·복원
- 전체 기록 초기화
- 저장 데이터 스키마 검증과 비정상 값 제한

### Phase 4 — PWA와 사용성

- Web App Manifest와 192px/512px 아이콘
- 앱 셸과 메뉴 이미지 전체를 캐시하는 Service Worker
- 네트워크 상태 표시 및 오프라인 내비게이션 대체
- Web Share API와 클립보드 복사 대체
- 키보드 포커스, `aria-*`, 대화상자 역할, 축소 모션 지원
- 설치 바로가기의 빠른 추천 쿼리 처리

### Phase 5 — 검증 전략

- Node 내장 테스트 러너로 외부 개발 의존성 없이 실행
- 추천 점수, 개인화 감점, 저장소 정제, 이미지·캐시 경로 무결성 검사
- Playwright/Chromium으로 모바일과 데스크톱에서 실제 클릭 흐름 검증
- 브라우저 콘솔 오류, 실패한 네트워크 요청, localStorage 기록 여부 확인
- 화면 캡처를 통한 레이아웃 검수

### Phase 6 — 배포 전략

- 소스와 계획을 `main` 브랜치에 보관
- 사용자 인증으로 `main`을 푸시하고, `npm run build`로 만든 `dist/` 전용 커밋을 `gh-pages`에 직접 푸시
- REST API를 통해 `gh-pages` 루트(`/`)를 Pages 게시 소스로 생성 또는 갱신
- `.nojekyll`로 Jekyll 처리 우회
- 배포 완료 상태를 확인한 뒤 실제 URL에서 HTML, JavaScript, 메뉴 이미지와 Manifest를 재검증
- 배포 산출물 구성을 `scripts/build-pages.mjs`로 결정론적으로 만들고 전체 절차를 `scripts/publish-pages.sh`로 자동화

## 5. 후속 개선 후보

MVP 이후 실제 사용 기록을 바탕으로 메뉴 가중치를 조정하고, 사용자가 브라우저에서 직접 메뉴와 사진을 추가하는 IndexedDB 기반 기능을 별도 페이즈로 확장할 수 있다. 이 기능도 서버 전송 없이 구현한다.
